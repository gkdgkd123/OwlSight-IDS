# Redis Guidelines (State Hub)

> Redis patterns, key design, and data conventions for OwlSight-IDS.

---

## Overview

OwlSight-IDS uses Redis as the **central state hub** and **message broker**.
There is no SQL database — Redis stores all flow state, inter-module messages,
and serves as the async task queue for LLM analysis.

---

## Role in Architecture

| Function | Redis Feature | Usage |
|----------|---------------|-------|
| Flow state storage | Hash (`HSET/HGETALL`) | Per-flow metadata (scores, decisions, LLM results) |
| Module communication | Pub/Sub (`PUBLISH/SUBSCRIBE`) | Suricata Early Abort signals |
| LLM task queue | List (`LPUSH/BRPOP`) | Producer-consumer pattern for LLM tasks |
| Failed task tracking | List (`LPUSH`) | `llm_failed_queue` for exhausted retries |

---

## Key Naming Convention

### Flow Keys (Five-Tuple)

Format: `{src_ip}:{src_port}-{dst_ip}:{dst_port}-{PROTO}`

```python
# Generated by src/utils.py
def generate_five_tuple_key(src_ip, src_port, dst_ip, dst_port, protocol):
    return f"{src_ip}:{src_port}-{dst_ip}:{dst_port}-{protocol}"

# Example: "192.168.1.100:54321-10.0.0.50:80-TCP"
```

Router scans for keys matching pattern `*:*-*:*-*`.

### Queue Names

| Name | Type | Producer | Consumer |
|------|------|----------|----------|
| `llm_task_queue` | List | Module 3 (Router) | Module 4 (LLM) |
| `llm_failed_queue` | List | Module 4 (LLM retry) | Manual inspection |

### Pub/Sub Channels

| Channel | Publisher | Subscriber |
|---------|-----------|------------|
| `suricata_alerts_channel` | Module 1 (Suricata) | Module 2 (Early Flow) |

## Hash Field Schema

### Flow State Hash

All fields are stored as strings (Redis Hash limitation):

| Field | Type | Set By | Description |
|-------|------|--------|-------------|
| `suricata_alert` | `"true"/"false"` | M1 | Whether Suricata flagged this flow |
| `signature` | string | M1 | Suricata rule name |
| `severity` | int string | M1 | Suricata severity level |
| `xgb_score` | float string | M2 | XGBoost prediction (0-1) |
| `anomaly_score` | float string | M2 | Isolation Forest anomaly (0-1) |
| `packet_count` | int string | M2 | Packets captured at inference time |
| `flow_start_time` | float string | M2 | Unix timestamp of first packet |
| `features` | JSON string | M2 | 18-dim feature vector |
| `decision` | enum string | M3 | BLOCK/PASS/ZERODAY_HUNT/LLM_ANALYZE |
| `decision_time` | float string | M3 | Unix timestamp of decision |
| `llm_result` | JSON string | M4 | Full LLM analysis result |
| `llm_verdict` | enum string | M4 | benign/suspicious/malicious/unknown |
| `llm_severity` | enum string | M4 | Critical/High/Medium/Low/Info |
| `llm_threat_type` | string | M4 | e.g., "Reconnaissance", "Successful_RCE" |
| `llm_is_successful_attack` | enum string | M4 | yes/no/unknown |
| `llm_is_malicious` | bool string | M4 | Derived from verdict |
| `llm_confidence` | float string | M4 | 0-1 confidence score |
| `llm_attack_type` | string | M4 | Backward-compatible alias |
| `llm_threat_level` | string | M4 | Backward-compatible alias |

## TTL Strategy

| Stage | TTL | Rationale |
|-------|-----|-----------|
| Initial write (M1/M2) | `config.redis.ttl` (default 60s) | Long enough for Router to scan |
| After decision (M3) | Refreshed to `config.redis.ttl` | Prevent expiry during LLM queue wait |
| After LLM result (M4) | 300s (5 min) | Results kept for downstream consumption |
| PASS decision (M3) | Immediate `DELETE` | Normal flows get cleaned up immediately |

## Connection Patterns

### Connection Factory

All modules use `RedisConnectionFactory` (singleton pool):

```python
from ..config.redis_factory import RedisConnectionFactory

# Shared pool: max_connections=50, timeout=5s, keepalive, health_check=30s
client = RedisConnectionFactory.get_client_with_retry(config)

# Dedicated client: for Pub/Sub and BRPOP (these commands hold connections)
dedicated = RedisConnectionFactory.get_dedicated_client(config)
```

### When to Use Dedicated vs Shared

| Scenario | Connection Type | Why |
|----------|----------------|-----|
| Normal HSET/HGET/SCAN | Shared pool | Frequent short operations |
| Pub/Sub `subscribe()` | Dedicated | Holds connection exclusively |
| BRPOP (blocking pop) | Shared (via existing client) | Blocks up to `timeout` seconds |
| Pipeline | Shared | Multiple operations on same connection |

---

## Pipeline Usage

Use pipelines for multi-field writes to guarantee atomicity:

```python
pipe = self.redis_client.pipeline()
pipe.hset(flow_key, "xgb_score", str(xgb_score))
pipe.hset(flow_key, "anomaly_score", str(anomaly_score))
pipe.hset(flow_key, "features", json.dumps(features))
pipe.expire(flow_key, self.redis_config.ttl)
pipe.execute()
```

**Rule**: Any write of 2+ fields to the same key MUST use a pipeline.

---

## SCAN vs KEYS

Always use `SCAN` for key iteration — `KEYS *` blocks Redis:

```python
cursor, keys = self.redis_client.scan(
    cursor=cursor,
    match="*:*-*:*-*",
    count=500
)
```

---

## Anti-Patterns

| Pattern | Why It's Wrong | Correct Approach |
|---------|---------------|-----------------|
| `redis.keys("*")` | Blocks entire Redis | `redis.scan(match=..., count=500)` |
| Multiple `hset()` without pipeline | Partial writes on failure | Use `pipeline()` |
| Creating connections in module code | Bypasses pool, DRY violation | Use `RedisConnectionFactory` |
| Hardcoded TTL values | Inconsistent expiry | Use `config.redis.ttl` |
| Pub/Sub error kills main flow | Pub/Sub is best-effort | Wrap in try-except, log and continue |
